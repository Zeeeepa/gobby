# Background & Motivation
Gobby currently maintains two separate knowledge representations in Neo4j:
- **Memory Graph:** Semantic concepts, user preferences, and high-level architectural constraints extracted from user conversations (using labels like `_Entity` and `Memory`).
- **Code Graph:** Concrete AST representations of the codebase generated by the indexer (using labels like `CodeSymbol` and `CodeFile`).

Currently, these two domains are siloed. Bridging these graphs would unlock natively context-aware retrieval. When an agent searches for a conceptual feature (like "authentication" or "the JWT system"), Gobby could instantly traverse the graph to return the semantic memory *and* the exact codebase symbols that implement it, dramatically reducing an agent's exploration time.

# Scope & Impact
This change will introduce a cross-graph relationship (`RELATES_TO_CODE`) originating from `_Entity` nodes and terminating at `CodeSymbol` nodes. This relationship will be automatically mapped in the background when new memories are created and ingested.

# Proposed Solution
Instead of forcing an LLM to hallucinate code symbols or read the entire codebase, we will utilize the existing `VectorStore` (Qdrant) to semantically match extracted memory entities to the existing `gobby-code` symbol index.

## Implementation Steps
1. **Pass Project Context:** Update `KnowledgeGraphService.add_to_graph` to accept an optional `project_id` parameter. Update callers in `MemoryManager` to pass the `project_id` from the source `MemoryRecord`.
2. **Semantic Matching (Step 9):** Add a new step to the memory ingestion pipeline in `KnowledgeGraphService`:
   - If `project_id` is present, take the entity embeddings that were just generated in Step 7.
   - For each entity, perform a vector search against the `gobby_code_{project_id}` Qdrant collection using the `self._vector_store.search()` wrapper.
   - Filter the results against a strict similarity threshold (e.g., `score > 0.82`) to prevent loose/noisy graph edges.
3. **Cross-Graph Linking:** Accumulate the matched symbols and write the relationships to Neo4j via a single batched Cypher query using `self._neo4j.query()`:
   ```cypher
   UNWIND $links AS link
   MATCH (e:_Entity {name: link.entity_name})
   MATCH (c:CodeSymbol {id: link.symbol_id, project: $project_id})
   MERGE (e)-[r:RELATES_TO_CODE]->(c)
   SET r.score = link.score
   ```
4. **Graceful Degradation:** Ensure the search gracefully handles cases where the `gobby_code_{project_id}` collection does not exist (e.g., if the user has not yet initialized the code indexer for the project).

# Verification & Testing

- Ensure no regressions in memory extraction speed (since vector searches against local Qdrant take single-digit milliseconds).
